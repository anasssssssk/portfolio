Project Deep Dive - UniNa Corse Formula Student

Teaching our driverless race car to know exactly where it is

On a driverless Formula Student car, the controller only steers as well as the position it's given. GPS is unreliable near buildings and unusable underground, exactly where our car still needs to know where it is. I built a ROS 2 localisation module that solves this with Adaptive Monte Carlo Localisation (AMCL), and it now feeds position directly into the car's controller with no drift versus the team's SLAM-based position, even over extended runs.

The Problem

Our autonomous car plans every turn and brake point based on knowing exactly where it is on track. Wheel encoders and IMUs drift over time, GPS drops out near buildings and doesn't work underground, and even our SLAM-based position built live while driving can accumulate small errors over a run. The controller needs one clean, trustworthy position estimate to steer by, not several conflicting ones.

The Approach

AMCL solves this with a "particle filter". The car maintains thousands of hypotheses about where it might be, each a guess at position and heading. I built this as a full ROS 2 module in C++: it takes in cone detections from our perception system, plus steering angle and speed from either wheel encoders or an IMU, and propagates each hypothesis forward using a kinematic bicycle model which is the same simplified physics used to predict how the car actually turns.

Each hypothesis is then scored against which cones it "expects" to see versus what perception actually detected on track. Good hypotheses survive and multiply; poor ones are pruned. I added a health check (effective sample size) that detects when the filter is becoming unreliable and automatically resamples to recover, keeping the estimate stable rather than letting it silently degrade mid-run.

Against plain odometry, wheel encoders or IMU alone, this is a clear win. Odometry has nothing to correct itself against, so small errors accumulate every timestep and the position estimate drifts steadily worse over a run. MCL corrects itself against the map on every update, so it doesn't carry that accumulating error.

I also built the module to run two ways, both feeding the same core filter: live, using the map our SLAM system builds in real time on track, or offline, using a static CSV map for controlled bench testing the same flexible pattern already used for switching between IMU and Wheel encoder input. That let the team validate and tune the filter safely off-car before trusting it on-track.

The Limitation — and Why It's Worth It Here

MCL isn't free of tradeoffs. Unlike SLAM, which builds a map while also estimating position from it, MCL can only localise against a map it already has, it can't build one itself, and it doesn't work at all against an unknown track. In our pipeline, that map comes from SLAM's own output or from a static CSV recorded earlier, so MCL is only ever as good as the map it's handed.

That's a real constraint, but it fits Formula Student well: our track layout is known and mappable before the car ever needs to drive it autonomously at speed, whether from a prior SLAM pass (Trackdrive) or a pre-recorded CSV (Skidpad). Once MCL has that map, it only has to solve localization not mapping and localisation together the way SLAM does so its position estimate held steadier and more accurate than the team's own live SLAM position over extended runs, with no drift. In a competition setting where the map is knowable in advance, that tradeoff is a straightforward win.

Vs. GPS

This is also where MCL wins over GPS. It never depends on an external signal at all — instead of listening for satellites, it continuously checks what the car's own sensors actually see against the known track layout, and corrects itself in real time. GPS isn't just "unreliable near buildings" — it's multipath reflections off grandstands and trackside structures that throw the position off by metres, and complete signal loss underground where there's no line of sight to satellites at all. MCL doesn't have either failure mode, since it isn't listening for a signal to begin with.

Results

Before touching the physical car, the logic was validated in MATLAB/Simulink against modelled vehicle dynamics, reducing the risk of costly on-track errors. On the real car, this AMCL module now feeds the controller's position input directly. Tested both offline against a static CSV map and live against our own SLAM map, it held a steady, accurate position with no drift versus the team's SLAM position, even over extended runs.

Engineering Practice

Beyond the core algorithm, I built the module to be something the team could actually operate: live trajectory and particle-cloud visualisation, projected-cone debugging overlays, smoothed pose output using a One Euro filter, and structured diagnostics so the system was easy for the rest of the team to monitor and trust, not just mathematically correct. A meaningful part of the work was debugging initialisation and runtime stability across our different startup modes, and restructuring the node so new features could be merged without breaking what already worked.
